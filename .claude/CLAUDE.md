# User-scope instructions

## 1. 언어
- 기본 답변은 **한국어**로 한다.
- 단, 사용자가 영어로 요청하면 영어로 답변한다.

## 2. Git
- **`main` 브랜치에 직접 커밋하지 않는다.** (절대 금지)
- 커밋이 필요하면 항상 별도의 브랜치를 새로 만들어서 커밋한다.
- 현재 브랜치가 `main`이면, 커밋 전에 먼저 새 브랜치를 생성한다.

# 코딩 행동 지침

LLM이 자주 저지르는 코딩 실수를 줄이기 위한 행동 지침. 필요에 따라 프로젝트별 지침과 함께 적용한다.

**트레이드오프:** 이 지침은 속도보다 신중함에 무게를 둔다. 사소한 작업에는 판단에 따라 유연하게 적용한다.

## 3. 코딩 전에 생각하기

**넘겨짚지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현하기 전에:
- 전제를 명시적으로 밝힌다. 불확실하면 물어본다.
- 해석이 여러 가지라면 모두 제시한다 — 조용히 하나만 고르지 않는다.
- 더 단순한 방법이 있으면 말한다. 타당하다면 반론을 제기한다.
- 불분명한 점이 있으면 멈춘다. 무엇이 헷갈리는지 짚고, 물어본다.

## 4. 단순함 우선

**문제를 해결하는 최소한의 코드만. 추측성 코드는 금지.**

- 요청받지 않은 기능은 만들지 않는다.
- 한 번만 쓰는 코드에 추상화를 만들지 않는다.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 넣지 않는다.
- 일어날 수 없는 상황에 대한 에러 처리를 하지 않는다.
- 200줄을 썼는데 50줄로 가능하다면, 다시 쓴다.

스스로 물어볼 것: "시니어 엔지니어가 이걸 보고 과하다고 할까?" 그렇다면 단순화한다.

## 5. 최소 변경 (Surgical Changes)

**꼭 필요한 것만 건드린다. 내가 만든 것만 정리한다.**

기존 코드를 수정할 때:
- 주변 코드, 주석, 포매팅을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 내 취향과 다르더라도 기존 스타일을 따른다.
- 무관한 죽은 코드를 발견하면 언급만 한다 — 삭제하지 않는다.

내 변경으로 고아 코드가 생겼을 때:
- 내 변경으로 쓰이지 않게 된 import/변수/함수는 제거한다.
- 요청받지 않는 한, 원래부터 있던 죽은 코드는 제거하지 않는다.

기준: 변경한 모든 줄은 사용자의 요청과 직접 연결되어야 한다.

## 6. 목표 기반 실행

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 바꾼다:
- "검증 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 통과시킨다"
- "X 리팩터링" → "리팩터링 전후로 테스트가 통과하는지 확인한다"

여러 단계 작업은 간단한 계획을 먼저 밝힌다:
```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
```

강한 성공 기준이 있어야 독립적으로 반복할 수 있다. 약한 기준("돌아가게 만들어")은 계속 확인을 요구하게 만든다.
