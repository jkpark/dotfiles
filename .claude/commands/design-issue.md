당신은 현재 리포지토리의 이슈 트래킹을 돕는다. 사용자가 준 **이슈 설명**을 받아, 잘 구성된 GitHub 이슈를 계획하고 생성한다.

이슈 설명: $ARGUMENTS

## 목표

애매한 아이디어를 명확한 GitHub 이슈로 만든다. 무엇을 해결하려는지 불명확하면 **먼저 사용자에게 물어보고**, 명확해지면 표준 템플릿으로 이슈를 만든다.

## 실행 순서

### 1. 리포/호스트 확인
- `git remote get-url origin`으로 리모트 URL을 읽는다.
- 여기서 **호스트**와 **owner/repo**를 추출한다. (예: 리모트가 `github.sec.samsung.net/HDP/CarePlanManager`면 호스트 = `github.sec.samsung.net`, owner/repo = `HDP/CarePlanManager`)
- 이후 모든 `gh` 명령은 `GH_HOST=<호스트> gh ... --repo <owner/repo>` 형태로 실행한다. (엔터프라이즈 호스트일 수 있으므로 `GH_HOST`를 반드시 지정. 호스트가 `github.com`이면 그대로 두어도 된다.)

### 2. 의도 명확화 (불명확하면 반드시 질문)
이슈 설명이 다음을 판단하기에 충분한지 검토한다.
- **목적**: 무엇을 이루려는가 / 성공 기준은 무엇인가
- **범위**: 무엇이 포함되고 무엇이 제외되는가
- **배경**: 왜 지금 필요한가

부족하면 **AskUserQuestion으로 되묻는다.** 목적·범위가 명확해질 때까지 반복한다. 사용자가 추측해도 된다고 말하지 않는 한, 애매한 채로 이슈를 만들지 않는다.

### 3. 리포 컨텍스트 탐색
- 리포의 문서와 관련 소스를 훑어, 이슈 배경과 **참고 링크**(관련 문서·코드 경로)를 보강한다.
- 있으면 `README`, `docs/`, `AGENTS.md`/`CLAUDE.md`/`CONTRIBUTING.md` 같은 규약 문서를 우선 확인하고, 그 리포가 정한 문서/아키텍처 권위 규칙을 따른다. 없으면 소스 구조에서 직접 파악한다.

### 4. 유사 이슈 중복 검색
- `GH_HOST=<호스트> gh issue list --repo <owner/repo> --state all --search "<핵심 키워드>"`로 유사 이슈를 찾는다.
- 강하게 유사한 이슈가 있으면 **목록을 사용자에게 보여주고**, 다음 중 무엇을 할지 확인한다:
  - 새 이슈를 그대로 만든다
  - 기존 이슈에 코멘트를 단다 (그렇다면 코멘트 내용을 제안하고 확인 후 게시)
  - 중단한다
- 유사 이슈가 없으면 그대로 진행한다.

### 5. 이슈 초안 작성
아래 **표준 템플릿**으로 제목과 본문을 작성한다. 본문 언어는 이슈 설명의 언어에 맞춘다(기본 한국어). 각 섹션은 이슈 설명 + 3단계에서 모은 컨텍스트로 채운다. 해당 없는 항목은 억지로 채우지 말고 섹션 자체를 최소화한다.

```markdown
## 배경
<왜 이 이슈가 필요한지, 현재 상황>

## 목적
<이 이슈가 해결하려는 것과 성공 기준>

## 범위
- 포함:
- 제외:

## 작업 항목
- [ ] <실행 가능한 항목>
- [ ] ...

## 완료 조건
- <완료로 판단할 기준>

## 참고
- <관련 문서 경로, 링크, 유사 이슈 번호>
```

제목은 간결한 한 줄로 짓는다.

### 6. 초안 확인 후 생성
- 완성한 제목·본문 초안을 사용자에게 보여준다.
- **사용자 승인을 받은 뒤에만** 이슈를 생성한다 (이슈 생성은 외부 게시 행위다. 승인 없이 생성하지 않는다).
- 승인되면:
  ```
  GH_HOST=<호스트> gh issue create --repo <owner/repo> --title "<제목>" --body "<본문>"
  ```
- 생성된 이슈 URL을 사용자에게 보고한다.

## 하지 않는 것
- 코드 구현/변경, 브랜치 생성, commit·push는 하지 않는다. 이 명령은 **이슈 생성까지만** 한다.
- 라벨/담당자는 사용자가 요청할 때만 지정한다.
